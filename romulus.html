The Problem

I've worked across product, design, and engineering for a decade — startups in health, web3, energy, consumer software. The recurring constraint wasn't ideas or skill. It was throughput.

Strong opportunities surface faster than one person can evaluate and build them. Most AI workflows helped with isolated tasks but didn't preserve context well enough for real product work. Sessions reset. Notes fragmented. Good ideas disappeared into chat history. I needed a system that stayed on, retained project memory, and supported the full path from signal to shipped artifact — without removing judgment from the loop.

~/romulus/setup.md

The Setup

Romulus runs on OpenClaw and lives on a dedicated Mac mini named Roma Eterna. It's always on — running scheduled jobs, monitoring things in the background, and sending a morning brief to Discord every day at 8 AM.

Memory lives in an Obsidian vault: daily notes, project files, operating context, decisions, and lessons. Every session reads from it. Every session writes back. 83 files, 29 project notes, 31 daily logs, 11 key decisions — dating back to first boot on February 3, 2026.

~/romulus/decisions.md

Why I Built It This Way

💬
Discord as the interface
It's always in my pocket, supports threaded context, and gives recurring outputs a natural home. It also doubles as a lightweight log — every brief, every command, every response is timestamped and searchable.
📓
Obsidian as persistent memory
Markdown files are durable, inspectable, and not locked to a proprietary layer. I can review, restructure, and evolve memory without vendor risk. It was also a deliberate choice over the built-in session memory in OpenClaw — which is session-bound, not persistent.
🖥
Dedicated Mac mini over cloud
I wanted a machine that was always available, stable, and separate from my daily laptop workflow. A dedicated node means cron jobs run reliably, context is always loaded, and there's no cold-start latency. Simpler to reason about than a cloud VM.
🧠
Multi-model instead of one-model-for-everything
Different tasks have different cost/capability tradeoffs. I apply the same resource allocation thinking to AI that I'd apply to any system: match the tool to the job. Routing a morning brief through a $0.000001/token model instead of Opus saves real money at scale without any quality loss.
~/romulus/workflow.sh

The Workflow

Every project moves through the same four stages. The models change. The loop stays fixed.

01
Research
Grok queries X live — classifying signal from noise, surfacing recurring pain points, flagging what's trending versus what's overdone. The output is a ranked list of opportunities worth evaluating, not a raw feed.
02
Analysis
OpenAI models pressure-test feasibility: stack complexity, competitive landscape, build time, distribution path. Structured output. Forces a real decision on whether the thing is worth the week.
03
Synthesis
Both route back through Romulus, where project memory and prior decisions are loaded as context. The system retrieves relevant past work, compares options, flags contradictions. Final call stays with me.
04
Execution
Claude Code handles multi-file implementation. Romulus tracks Git state, Vercel deploys, and session context between builds. When one sprint ends and the next starts, nothing has to be re-explained.
~/romulus/models.json

Model Routing Philosophy

Not every task deserves the same model. I treat model selection the same way I'd treat any system design decision: match capability and cost to the job.

Minimax M2.7
The nervous system. Morning briefs, balance checks, scheduled tasks. Fast, cheap, reliable. Zero overhead.
Claude Sonnet
The thinking partner. Research synthesis, strategic discussions, things that need actual depth — not just output.
Claude Opus
The workhorse. Multi-file implementations, complex refactors, the builds that actually matter.

What I'd Tell Another Builder

The model is not the product. The architecture is. Persistent memory, explicit routing, a fixed workflow, and a stable host change what's possible. Without them, you have a fast chatbot. With them, you have a system.

Articulate your tradeoffs. Why this interface, why this memory substrate, why this model for this task — the decisions are where the product engineering shows. Anyone can list the tools. Fewer people can explain why they chose them over the alternatives.

Build something you actually use every day. The morning brief is the most useful thing in this system. Not because it's impressive — because I check it every morning before I leave. That's the real test.
