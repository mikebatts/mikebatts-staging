CASE STUDY • 2026

Building a Personal AI System
for Shipping Faster

How I designed and built an always-on AI agent — named after Rome's founding king — to close the gap between thinking and shipping.

01 — OVERVIEW

The Problem

I've spent the past decade in product — designing, building, and shipping across startups in health, web3, energy, and consumer apps. For most of that time, the bottleneck was me. Not for lack of skill or clarity, but bandwidth. Good ideas pile up fast. Building them alone is slow.

In early 2026 I started building Romulus — a personal AI system designed to close that gap. Named after Rome's founding king. Running on a Mac mini called Roma Eterna. This is a look at how it came together, what we shipped with it, and where it's heading.

02 — THE SETUP

Roma Eterna

Romulus runs on OpenClaw, a platform for persistent personal AI agents. It lives on a dedicated Mac mini (Roma Eterna) so it's always on — running cron jobs, monitoring things in the background, delivering a morning briefing to Discord every day at 8 AM.

The interface is Discord. The memory is an Obsidian vault: daily notes, project files, decisions, going back to the first boot session on February 3rd, 2026.

The goal was a system that compounds — one that gets more useful over time, not one you have to re-explain yourself to every session. 83 files of context, always loaded.

03 — THE WORKFLOW

How the Work Actually Runs

Every project follows the same loop:

01
Research
Grok surfaces real-time signal from X — what's getting traction, what people are complaining about, what's under-served right now.
02
Analysis
OpenAI Codex stress-tests the idea. Technical feasibility, stack options, competitive landscape. Structured thinking, fast.
03
Synthesis
Both come back to Romulus. This is where the work gets filtered. Ten years of product judgment is the actual filter — Romulus helps surface the data, the go/no-go is a human call.
04
Execution
Claude Code (Opus) handles the heavy build work. Romulus orchestrates, tracks, and keeps context between sessions. Git and Vercel handle deployment.

Model Routing Philosophy

Not every task needs the same tool. I apply the same resource allocation thinking to AI that I'd apply to any engineering system. Right tool, right job.

Minimax M2.7
The nervous system. Morning briefs, balance checks, scheduled tasks. Fast, cheap, reliable. Zero overhead.
Claude Sonnet
The thinking partner. Research synthesis, strategic discussions, things that need actual depth — not just output.
Claude Opus
The workhorse. Multi-file implementations, complex refactors, the builds that actually matter.
04 — WHAT WE SHIPPED

Projects Built with This System

Chronicle
thischronicle.com

A daily history guessing game. One event per day, three chances to guess the year. Wordle-style digit feedback. 90 hand-curated puzzles spanning five centuries.

Specced, scaffolded, and pushed to production in a focused sprint. The design came first — share card format, difficulty calibration, the no-clues Sunday mode. Claude Code built to the spec.

Next.js + Tailwind
Caveat
trycaveat.com

AI contract analyzer. Upload any PDF or DOCX, get a risk report in 60 seconds. Flags bad terms, missing clauses, unfavorable language. Built on Next.js 15, Stripe, GPT-4o. Privacy-first — contracts are never stored.

The core insight came from genuine signal: contract review is expensive and slow, and most people skip it. The product addresses a real friction point that freelancers, contractors, and founders face every day.

Next.js 15 + Stripe + GPT-4o
The Morning Brief
delivered daily to Discord

Every morning at 8 AM: weather for Park Slope, subway departures for the R train from Prospect Ave to Canal St, and a product signal digest. All three in Discord. Custom Node.js script pulling live MTA GTFS-RT protobuf feeds.

Built something I actually use every day. Real utility, not a demo. The kind of tool that makes the system feel worth it before breakfast.

05 — THE SYSTEM

The Memory Architecture

The Obsidian vault is the part of this system I'm most proud of.

83 files. 12 core memory documents. 29 project notes. 31 daily logs. 11 key decisions. Every session reads from it. Every session writes to it. Romulus knows the context of every project, every past decision, every lesson learned — not because I repeat myself, but because the system holds it.

This is what makes it compound. Each project makes the next one faster. The vault grows more useful over time. The system learns from itself.

06 — WHAT'S NEXT

Legion

The next layer is already designed. Legion is an orchestration system — Romulus as conductor, multiple Claude Code workers running in parallel with git worktree isolation.

Task decomposition, parallel execution, quality gates before merge, results synthesized and delivered to Discord. The goal is to move from one person building with an AI partner to something closer to a small studio.

Phase 1: MVP Swarm — 2-3 parallel workers, collected results, completion notification.
Phase 2: Worktree isolation — each worker in its own git branch.
Phase 3: Quality gates — automated tests before merge.
Phase 4: Knowledge accumulation — past learnings prime future tasks.

"Romulus commands Legion." — Roman military structure. A disciplined coordinated force under single command.
07 — TAKEAWAYS

What I'd Tell Another Builder

The system matters as much as the tools. Getting the infrastructure right — the memory architecture, the model routing, the always-on setup — is what makes everything else go faster.

Good product judgment still applies. AI amplifies the work, it doesn't replace the thinking. Knowing what not to build is just as important as how fast you can ship.

The best measure of a system like this is whether you actually use it. Romulus runs every day. It delivers the morning brief. It keeps context across weeks of work. It's infrastructure now, not an experiment.
